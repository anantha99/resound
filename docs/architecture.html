  • Red = external systems (sources + providers behind OpenRouter).
  • Yellow = config files the operator edits.
  • Blue = pipeline layers (1, 2, 3, 5).
  • Purple = LLM gateway — every dotted "LLM call" terminates here.
  • Green = memory layer.
  • Pink = dashboard.
  • Solid arrows = data flow. Dotted = config / prompts / observability. Thick (==>) = writes to memory.

LLM Gateway internals — one classify call

What happens inside the gateway for a single classification, including the retry/fallback chain.

sequenceDiagram
    autonumber
    participant C as OpenRouterClassifier
    participant GW as OpenRouterGateway
    participant CFG as models.yaml
    participant OR as OpenRouter API
    participant P1 as Primary model
(claude-sonnet-4-6)
    participant P2 as Fallback 1
(gpt-4.1)
    participant P3 as Fallback 2
(gemini-2.5-pro)
    participant DB as llm_calls table

    C->>GW: complete(stage="classify", prompt, schema)
    GW->>CFG: lookup classify stage
    CFG-->>GW: model=claude-sonnet-4-6,
fallbacks=[gpt-4.1, gemini-2.5-pro]

    GW->>OR: POST /chat/completions
model=claude-sonnet-4-6
response_format=json_object
    OR->>P1: route request
    P1--xOR: 503 Service Unavailable
    OR-->>GW: 503

    Note over GW: Transient error,
retry with backoff (1s)
    GW->>OR: retry (attempt 2)
    OR->>P1: route request
    P1--xOR: 503
    OR-->>GW: 503

    Note over GW: Retry exhausted,
move to fallback chain
    GW->>OR: POST /chat/completions
model=gpt-4.1
    OR->>P2: route request
    P2-->>OR: 200 OK + JSON
    OR-->>GW: response
usage={tokens, cost}

    Note over GW: Validate JSON
against schema
    GW->>DB: INSERT llm_calls
stage=classify
model=gpt-4.1
tokens, cost, latency
fallback_used=true
    GW-->>C: LLMResponse(content,
model_used=gpt-4.1,
cost_usd, latency_ms)
Key behaviors
  • Single chokepoint — classifier never sees model slugs.
  • Config-driven — model resolved from models.yaml at call time.
  • Transient (5xx, 429) retries on same model; other 4xx skip to fallback.
  • Every attempt logged, including failed primary, for fallback-rate telemetry.
  • If full chain fails, LLMGatewayError raised; signal recorded unclassified.

Data model

Append-only memory schema. Nothing mutates except feedback acknowledgement flags.

erDiagram
    SIGNALS ||--|| CLASSIFICATIONS : "1:1 (when classified)"
    SIGNALS ||--o| ROUTES : "0:1 (after routing)"
    CLASSIFICATIONS ||--o| ROUTES : "0:1"
    ROUTES ||--o{ FEEDBACK_EVENTS : "many"
    ROUTES }o--o| OUTCOMES : "many:one"
    CLASSIFICATIONS ||--|{ LLM_CALLS : "1+ per classification"

Why this shape
  • signals is the immutable root — every later table FKs to it.
  • classifications carries model audit fields inline so the dashboard avoids joins for cost/quality views.
  • llm_calls is the granular ledger — one row per invocation, including failed primaries that triggered fallbacks.
  • outcomes is many-to-one with routes — one ship/fix can resolve many signals.
  • No mutation paths in v1 except feedback acknowledgement.
